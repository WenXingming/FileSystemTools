# 第 2 层：硬链接折叠与顺序读优化

## 优化背景与核心思路

在第 1 层按文件大小进行候选筛选之后，`dedup` 已经可以排除掉大小唯一的文件，只保留 size 发生碰撞的候选文件。此时流程可以抽象为：

```text
扫描目录
  -> 主线程收集文件路径、大小等元信息
  -> 主线程按 size 分组，大小唯一的文件直接跳过
  -> 线程池对 size 相同的候选文件计算内容 hash
  -> 主线程按 hash 分组
  -> 主线程输出重复文件组
```

但是，这里还有一个来自文件系统层面的优化空间：**路径不同，不一定代表物理文件不同**。

在类 Unix 文件系统中，多个路径可能通过硬链接指向同一个物理文件。也就是说，它们在目录树中看起来是多个文件路径，但底层实际共享同一份物理数据。如果不识别这种情况，程序可能会对同一份物理数据重复读取、重复 hash，从而产生冗余 I/O 和冗余计算。因此，在进入真正的内容 hash 之前，可以先利用文件系统元信息做一次物理层面的折叠。

硬链接折叠不会影响去重结果的正确性。原因是：如果两个路径拥有相同的 `(device, inode)`，它们指向同一个物理文件，也就共享同一份文件内容。因此，这些路径对应的 hash 结果必然相同。所以我们只需要对代表文件计算一次 hash，然后在结果阶段把 follower 展开即可。

这一层优化主要做两件事：

```text
硬链接折叠：
  通过 (device, inode) 识别同一个物理文件
  多个路径指向同一物理文件时，只选择一个代表文件参与 hash

顺序读优化：
  对小文件按 (device, inode) 排序
  尽量提高文件访问局部性，减少随机访问带来的额外开销
```

这里需要注意，硬链接折叠不是把文件从最终结果中删除，而是暂时减少 hash 阶段的物理工作量。

对于多个路径指向同一个 `(device, inode)` 的情况：

```text
第一次遇到：
  作为代表文件进入后续 hash 流程

后续遇到：
  作为 hardlink follower 暂存起来
  不重复读取和 hash

hash 完成后：
  将 follower 展开回结果集
  保留所有路径的去重语义
```

因此，这一步更准确地说是：折叠重复的物理 I/O，而不是丢弃文件路径。

## 使用 device + inode 判断硬链接

在文件系统中，`path` 是文件在目录树中的名字，而 `inode` 更接近文件本体。对于同一个文件系统上的普通文件，一个 inode 通常对应一份文件元数据和文件内容；如果多个路径拥有相同的 `(device, inode)`，就可以认为它们指向同一个物理文件。

因此，我在 `FileInfo` 中补充了文件系统元信息：

```cpp
struct FileInfo {
    std::string path;
    uint64_t size;
    uint64_t device;
    uint64_t inode;
};
```

其中：

```text
device:
  标识文件所在的设备或文件系统

inode:
  标识该设备上的文件对象
```

单独使用 inode 还不够严谨，因为不同设备上的 inode 编号可能重复。因此，需要使用 `(device, inode)` 共同作为物理文件身份。

## 硬链接折叠的实现

在 size collision 之后，程序遍历候选文件集合，用 `(device, inode)` 作为 key 进行折叠。

核心数据结构包括：

```cpp
std::vector<FileInfo> uniquePhysicalFiles;

std::map<std::pair<uint64_t, uint64_t>, std::vector<FileInfo>> hardlinkFollowers;
```

其中：

```text
uniquePhysicalFiles:
  保存每个物理文件的代表路径
  后续只对这些代表文件继续进行 partial hash 或 full hash（类似于过滤）

hardlinkFollowers:
  保存被折叠掉的其他路径
  key 是 (device, inode)
  value 是指向同一物理文件的 follower 路径列表
```

核心逻辑如下：

```cpp
std::vector<FileInfo> uniquePhysicalFiles;
std::map<std::pair<uint64_t, uint64_t>, std::vector<FileInfo>> hardlinkFollowers;

for (const FileInfo& file : sizeCollisions) {
    std::pair<uint64_t, uint64_t> devIno =
        std::make_pair(file.device, file.inode);

    if (hardlinkFollowers.find(devIno) == hardlinkFollowers.end()) {
        uniquePhysicalFiles.push_back(file);
        hardlinkFollowers[devIno] = std::vector<FileInfo>();
    } else {
        hardlinkFollowers[devIno].push_back(file);
    }
}
```

这段逻辑的含义是：

```text
如果第一次遇到某个 (device, inode)：
  说明这是该物理文件的代表路径
  放入 uniquePhysicalFiles，参与后续 hash

如果已经见过这个 (device, inode)：
  说明这是同一物理文件的另一个路径
  不再重复 hash，只记录到 hardlinkFollowers
```

经过这一步后，后续 hash 阶段处理的不再是所有路径，而是所有唯一物理文件。

## 硬链接展开的实现

硬链接折叠不会影响去重结果的正确性。原因是：如果两个路径拥有相同的 `(device, inode)`，它们指向同一个物理文件，也就共享同一份文件内容。因此，这些路径对应的 hash 结果必然相同。所以，我们只需要对代表文件计算一次 hash，然后在结果阶段把 follower 展开即可。

展开逻辑可以抽象为：

```text
对每个代表文件的 HashResult：
  找到相同 (device, inode) 的 follower 列表
  复制代表文件的 hash 结果
  替换其中的 FileInfo 为 follower
  追加回最终 hash 结果集合
```

对应代码结构如下：

```cpp
std::vector<HashResult> unfoldedHashes = fullHashes;

for (const HashResult& result : fullHashes) {
    std::pair<uint64_t, uint64_t> devIno =
        std::make_pair(result.file.device, result.file.inode);

    auto followerIt = hardlinkFollowers.find(devIno);
    if (followerIt != hardlinkFollowers.end()) {
        for (const FileInfo& follower : followerIt->second) {
            HashResult copied = result;
            copied.file = follower;
            unfoldedHashes.push_back(copied);
        }
    }
}
```

这里的关键点是：

```text
hash 阶段：
  follower 不重复参与 I/O 和 hash

结果阶段：
  follower 被重新展开回 HashResult

最终分组：
  所有路径仍然参与 find_exact_duplicates()
```

因此，硬链接折叠只是优化执行过程，不会改变最终输出语义。

## 顺序读优化

除了硬链接折叠之外，这一层还做了一个文件访问顺序优化：对文件候选集合按照 `(device, inode)` 排序。代码结构如下：

```cpp
std::sort(
    smallCandidates.begin(),
    smallCandidates.end(),
    [](const FileInfo& left, const FileInfo& right) {
        if (left.device != right.device) {
            return left.device < right.device;
        }
        if (left.inode != right.inode) {
            return left.inode < right.inode;
        }
        return left.path < right.path;
    }
);
```

这个优化的动机是：大量小文件场景下，单个文件读取的数据量很小，但文件打开、定位和读取的固定成本很高。如果访问顺序完全随机，真实磁盘上可能会产生较差的访问局部性。

按照 `(device, inode)` 排序后，程序会尽量按照文件系统中的物理身份顺序处理小文件，目标是改善访问局部性，降低 HDD 等存储设备上随机访问带来的额外开销。

需要注意，这里不能绝对地说“inode 排序一定触发顺序读”。不同文件系统、不同存储设备、Page Cache 状态、文件分配策略都会影响最终效果。因此，更严谨的表述是：

> inode 排序是一种尝试改善 I/O 局部性的工程优化，在冷缓存、真实磁盘、大量碎片小文件场景下更可能体现价值。

也就是说，这里的“顺序读优化”并不是保证物理磁盘一定按顺序读取，而是通过更合理的访问顺序，尽量减少完全随机访问带来的额外开销。

## 整体流程

加入这一层之后，整体流程可以变成：

```text
扫描目录
  -> 收集 path / size / device / inode 等元信息
  -> 第 1 层：按 size 分组，大小唯一直接跳过
  -> 第 2 层：按 (device, inode) 折叠硬链接
  -> 小文件按 (device, inode) 排序，尽量改善访问局部性
  -> 对唯一物理文件继续执行后续 hash 流程
  -> hash 完成后展开 hardlink follower
  -> 按 hash 分组
  -> 输出重复文件组
```

这一步在整个优化漏斗中的作用是：

```text
size 过滤：
  减少不可能重复的逻辑候选文件

hardlink 折叠：
  减少同一物理文件的重复 I/O 和重复 hash

inode 排序：
  尝试改善小文件读取的访问局部性
```

也就是说，size 过滤关注的是“哪些文件不可能重复”，而硬链接折叠关注的是“哪些路径其实指向同一份物理数据”。

## Benchmark 状态与边界

这一层优化在工程上是合理的，但当前 benchmark 还不能充分证明它的性能收益。

原因主要有两个。

第一，当前 benchmark 没有专门构造 hardlink-heavy 数据集。如果测试数据中没有大量硬链接，那么 hardlink 折叠自然不会体现明显收益。

第二，当前 benchmark 数据主要生成在 `/tmp` 下，容易受到 tmpfs、Page Cache 和热缓存影响。inode 排序这类 I/O 局部性优化，更适合在真实磁盘、冷缓存、大量碎片小文件场景下验证。

因此，这一层优化在报告中应该保持谨慎口径：

```text
硬链接折叠：
  工程上可以避免同一物理文件重复 hash
  但需要 hardlink-heavy benchmark 验证收益

inode 排序：
  工程上尝试改善小文件访问局部性
  但需要真实磁盘冷缓存 benchmark 验证收益
```

后续可以补充两个专项 benchmark：

```text
hardlink_heavy:
  构造大量 hardlink，验证 full hash 次数是否减少

cold_cache_many_small:
  在真实磁盘冷缓存下测试大量小文件，验证 inode 排序是否改善访问局部性
```

## 小结

第 2 层优化关注的是文件系统层面的重复工作和访问局部性。

硬链接折叠通过 `(device, inode)` 识别多个路径是否指向同一个物理文件。如果多个路径共享同一个物理文件，程序只对代表文件执行 hash，其他路径作为 follower 暂存，并在 hash 完成后展开回结果集。这样既减少了重复 I/O 和重复 hash，又保留了所有路径的输出语义。

顺序读优化则通过对小文件按 `(device, inode)` 排序，尝试改善真实磁盘上的访问局部性。它不是一个保证收益的算法优化，而是一个依赖文件系统、存储设备和缓存状态的工程优化。

因此，这一层的定位应该是：工程上合理、语义上安全，但性能收益需要通过 `hardlink_heavy` 和 `cold_cache_many_small` 等专项 benchmark 进一步验证。


Tips：本轮没有单独 benchmark。原因不是这个优化不重要，而是当前 benchmark 不适合验证它：

- 数据集没有专门构造 hardlink，无法体现硬链接折叠收益。
- benchmark 在 `/tmp` 生成数据，容易被 tmpfs 或 Page Cache 掩盖真实磁盘访问差异。
- inode 排序更适合在冷缓存、真实磁盘、大量碎片小文件场景验证。

后续应该新增：

- `hardlink_heavy`：大量 hardlink，验证 full hash 次数减少。
- `cold_cache_many_small`：真实磁盘冷缓存，验证 inode 排序效果。
