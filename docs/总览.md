# dedup 性能优化总结与 Benchmark 验证

## 文档定位

本文是 `dedup` 性能优化工作的总结文档，重点不是重复展开每一层优化的实现细节，而是回答三个问题：

```text
1. 整体优化思路是什么？
2. Benchmark 是如何设计和运行的？
3. 数据最终证明了哪些结论，哪些结论还需要进一步验证？
```

各个具体优化方法已经在前面的独立文档中展开，例如线程池并发执行、size 过滤、硬链接折叠、小文件固定成本优化、大文件 partial hash 漏斗、full hash 汇流与双轨调度等。

这篇文档主要负责把这些内容收束成一个完整的工程闭环。

---

## 总体优化思路

`dedup` 是一个本地文件去重工具，目标是扫描指定目录，找出内容相同的重复文件组。

最朴素的方案是：扫描所有文件，对每个文件计算完整内容 hash，然后按 hash 分组。这种方案简单直接，但在真实文件系统中并不高效。因为不同文件形态对应的性能瓶颈不同：

```text
大量小文件：
  单个文件内容很少，主要瓶颈往往是 open / close、buffer 分配、任务调度、future 管理等固定成本。

大文件：
  单个文件数据量大，主要瓶颈是完整读取和 full hash 带来的 I/O 成本。

文件系统层：
  多个路径可能通过 hardlink 指向同一个物理文件，如果不识别，会重复读取和重复 hash。

混合目录：
  小文件固定成本、大文件无效 I/O、任务调度负载均衡会同时出现。
```

因此，这轮优化不是简单地“加线程”或“换 hash 算法”，而是围绕文件形态建立成本模型，再根据成本模型设计分层处理流程。

整体优化路线可以概括为：

```text
扫描目录
  -> 收集 path / size / device / inode 等元信息
  -> 使用线程池建立并发 full hash 基线
  -> 按 size 分组，大小唯一的文件直接跳过
  -> 按 device / inode 折叠 hardlink，减少重复物理 I/O
  -> 按文件大小分成小文件轨道和大文件轨道

小文件轨道：
  -> 直接 full hash
  -> 重点降低每文件固定成本

大文件轨道：
  -> 先做 head + tail partial hash
  -> partial hash 不同则跳过 full hash
  -> partial hash 仍然碰撞才进入 full hash

最终：
  -> 小文件和大文件汇流到 full hash
  -> 展开 hardlink follower
  -> 按完整 hash 分组
  -> 输出重复文件组
```

可以用一句话总结：

> 线程池提升执行效率，size 和 inode 减少候选，小文件优化固定成本，大文件减少无效 I/O，最终 full hash 保证正确性。

### 机器环境

本轮 benchmark 在同一台机器上完成，主要环境如下：

| 项目                   | 值                                    |
| -------------------- | ------------------------------------ |
| 主机                   | `wxm-Precision-7920-Tower`           |
| CPU                  | 24 logical CPUs, 3500 MHz            |
| L1 Data Cache        | 32 KiB x12                           |
| L1 Instruction Cache | 32 KiB x12                           |
| L2 Cache             | 1024 KiB x12                         |
| L3 Cache             | 16896 KiB x1                         |
| Benchmark 日期         | 2026-06-25                           |
| 数据生成目录               | `/tmp/threadpool_dedup_bench_XXXXXX` |

需要注意，每次 benchmark 运行时都提示 CPU scaling enabled，因此本文更关注趋势和数量级，不把小幅波动解释为确定结论。

此外，benchmark 数据生成在 `/tmp` 下，容易受到 tmpfs、Page Cache、热缓存等因素影响。因此，当前结果更适合验证算法和工程优化方向，不完全等价于真实磁盘冷缓存场景。

### 构建方法

构建 benchmark 的命令如下：

```bash
cmake -S . -B build-bench \
  -DBUILD_BENCHMARKS=ON \
  -DCMAKE_BUILD_TYPE=Release \
  -DTHREADPOOL_SOURCE_DIR=/home/wxm/ThreadPool

cmake --build build-bench
```

其中：

```text
BUILD_BENCHMARKS=ON:
  开启 benchmark 构建

CMAKE_BUILD_TYPE=Release:
  使用 Release 模式，避免 Debug 构建影响性能结果

THREADPOOL_SOURCE_DIR:
  指向自定义线程池项目路径
```

---

### 运行方法

基础运行命令如下：

```bash
./build-bench/benchmark/dedup/dedupBenchmark \
  --benchmark_min_time=1.0 \
  --benchmark_repetitions=3 \
  --benchmark_report_aggregates_only=true
```

其中：

```text
--benchmark_min_time=1.0:
  每个 benchmark 至少运行约 1 秒，降低单次短运行噪声

--benchmark_repetitions=3:
  每个 workload 重复运行 3 次

--benchmark_report_aggregates_only=true:
  只输出聚合统计结果，重点关注 mean / median / stddev 等指标
```

如果需要保存机器可读结果，可以输出 JSON：

```bash
./build-bench/benchmark/dedup/dedupBenchmark \
  --benchmark_min_time=1.0 \
  --benchmark_repetitions=3 \
  --benchmark_report_aggregates_only=true \
  --benchmark_format=json \
  > benchmark/dedup/results/latest.json
```

## 指标口径

本文主要关注 Google Benchmark 输出中的 `Time mean`，也就是 wall time 的平均值。

| 指标                 | 含义                           |
| ------------------ | ---------------------------- |
| `Time`             | wall time，用户真实等待时间，是本文主要关注指标 |
| `CPU`              | CPU 实际执行时间                   |
| `_mean`            | 多次重复运行的平均值                   |
| `bytes_per_second` | Google Benchmark 按逻辑数据量计算的吞吐 |
| `items_per_second` | 每秒处理文件数                      |

选择 `Time` 作为主要指标，是因为 `dedup` 是一个面向用户的文件处理工具，用户更关心实际等待时间，而不是单纯的 CPU 时间。

需要注意，V4 引入 partial hash 后，`bytes_per_second` 的含义会发生变化。程序可能只读取大文件头尾几 KiB，但 Google Benchmark 仍可能按完整文件大小计算逻辑吞吐。因此，在 partial hash 场景下，`bytes_per_second` 不能直接理解成真实磁盘吞吐，更适合看 `Time mean` 的变化。

## Benchmark 设计

文件去重的瓶颈和文件形态强相关，因此 benchmark 不能只设计一种 workload。本文将 benchmark 分成基础 workload 和专项 workload。

基础 workload 用来观察整体吞吐变化，专项 workload 用来验证具体优化点。

| Workload                                     |   文件数 |    文件大小 | 重复比例 | 目的                       |
| -------------------------------------------- | ----: | ------: | ---: | ------------------------ |
| `100_files_4KiB`                             |   100 |   4 KiB |  20% | 小规模 sanity check         |
| `1000_files_4KiB`                            |  1000 |   4 KiB |  20% | 小文件基础吞吐                  |
| `10000_files_4KiB`                           | 10000 |   4 KiB |  20% | 大量小文件固定成本                |
| `1000_files_256KiB`                          |  1000 | 256 KiB |  20% | 中等文件吞吐                   |
| `many_small_unique_same_size`                | 10000 |   4 KiB |   0% | size 过滤失效时的小文件压力         |
| `many_small_duplicates`                      | 10000 |   4 KiB | 100% | 小文件重复场景                  |
| `large_same_size_different_content`          |    20 |   8 MiB |   0% | 验证非重复大文件 partial hash 收益 |
| `large_same_size_same_prefix_different_tail` |    20 |   8 MiB |   0% | 验证 head + tail 设计是否有效    |
| `large_duplicates`                           |    20 |   8 MiB | 100% | 验证真实重复大文件必须 full hash    |
| `mixed_realistic`                            |  1000 |      混合 |  20% | 模拟真实混合目录                 |

这些 workload 对应不同问题：

```text
小文件 workload：
  验证固定成本是否成为瓶颈。

大文件同大小但内容不同：
  验证 partial hash 是否能减少无效 I/O。

大文件同前缀但尾部不同：
  验证 head + tail 是否比只读 head 更稳。

真实重复大文件：
  验证最终仍然必须 full hash，不能因为 partial hash 破坏正确性。

混合目录：
  验证优化组合在更接近真实场景下是否仍然有净收益。
```

## 核心 Benchmark 结论

### 1. 线程池证明并发 full hash 有效

V1 已经是多线程 full hash 基线，而不是单线程基线。通过 1 / 2 / 4 / 8 线程对比，可以看到文件 hash 阶段具有明显并行性。

典型结果：

```text
1000_files_256KiB:
  1 线程 852 ms
  8 线程 123 ms
  约 6.93x 加速
```

这说明文件读取和 hash 计算之间存在可以并行化的空间。线程池能通过多个 worker 重叠 I/O 等待和 CPU 计算，从而降低 wall time。

但线程池只解决“如何更快地执行 hash”，不解决“哪些文件不需要 hash”。因此，它只是并发执行基线，后续还需要过滤和成本模型优化。

---

### 2. 小文件优化的主要收益来自 buffer 固定成本

小文件场景下，最稳定、最明显的收益来自 V2 的 buffer zero-initialization 优化。

原实现使用：

```cpp
std::vector<char> buffer(bufferSize_);
```

这会分配并初始化整块 buffer。但后续 `read()` 会覆盖 buffer 内容，因此提前清零是无意义成本。

优化后改为：

```cpp
std::unique_ptr<char[]> buffer(new char[bufferSize_]);
```

只分配，不做无意义清零。

8 线程下，基础 workload 对比如下：

| Workload            |      V1 |       V2 |      提升 |
| ------------------- | ------: | -------: | ------: |
| `100_files_4KiB`    | 2.41 ms | 0.749 ms | 约 3.22x |
| `1000_files_4KiB`   | 22.1 ms |  6.83 ms | 约 3.24x |
| `10000_files_4KiB`  |  214 ms |  70.2 ms | 约 3.05x |
| `1000_files_256KiB` |  123 ms |  32.1 ms | 约 3.83x |

这说明：大量小文件下，瓶颈不一定是 hash 算法本身，而可能是每个文件都会发生的固定成本。V2 的改动很小，但它命中了真实瓶颈，因此收益稳定。

---

### 3. 小文件 batch 没有被当前 benchmark 证明，但暴露了调度边界

V3 尝试把：

```text
1 个文件 = 1 个任务 = 1 个 future
```

改成：

```text
一批文件 = 1 个任务 = 1 个 future
```

这个优化的目标是减少任务提交和 future 管理成本。

但从 benchmark 看，小文件 batch 在当前环境下没有明显收益：

```text
10000_files_4KiB:
  V2 70.2 ms
  V3 70.1 ms
  基本持平

many_small_unique_same_size:
  V2 117 ms
  V3 119 ms
  略慢

many_small_duplicates:
  V2 121 ms
  V3 137 ms
  略慢
```

因此，不能把 V3 讲成一次成功的小文件提速优化。它更重要的价值是暴露了策略边界：batch 可以尝试服务于小文件，但不能无差别应用到大文件。

V3 将 batch 统一应用到大文件后，导致大文件任务出现严重长尾：

```text
large_duplicates:
  V2 77.4 ms
  V3 516 ms
```

这说明一个 worker 如果一次拿到多个大文件，会长时间占用线程，导致其他线程空闲等待，破坏负载均衡。

---

### 4. 大文件 partial hash 对非重复大文件非常有效

V4 的核心优化是 head + tail partial hash。它针对的是“大小相同但内容不同的大文件”。

设计逻辑是：

```text
size 相同的大文件
  -> 先读取 head 2KiB + tail 2KiB
  -> 计算 partial hash
  -> partial hash 不同，跳过 full hash
  -> partial hash 相同，再进入 full hash
```

这个设计的目标是减少无效 I/O。

以 V2 作为稳定基线，V4 的大文件专项结果非常明显：

| Workload                                     |      V2 |      V4 | 结论      |
| -------------------------------------------- | ------: | ------: | ------- |
| `large_same_size_different_content`          | 76.9 ms | 1.93 ms | 约 39.8x |
| `large_same_size_same_prefix_different_tail` | 77.1 ms | 1.97 ms | 约 39.1x |
| `large_duplicates`                           | 77.4 ms | 78.5 ms | 基本持平    |

这组结果说明：

```text
非重复大文件：
  partial hash 可以快速排除，避免 full hash，因此提升巨大。

前缀相同但尾部不同的大文件：
  head + tail 能有效过滤，说明只读 head 不够稳。

真实重复大文件：
  仍然必须 full hash，因此不会天然变快。
```

这个结果非常符合预期，也证明 partial hash 优化的是“无效 I/O”，不是所有大文件。

---

### 5. V4 的收益必须区分“修复回归”和“真实净收益”

报告中最容易误读的是 V3 到 V4 的巨大提升。

V3 的大文件退化来自 batch 策略误用于大文件。V4 同时做了两件事：

```text
1. 修复大文件 batch 长尾问题：
   大文件 full hash 使用 Batch Size = 1。

2. 引入 head + tail partial hash：
   非重复大文件可以提前跳过 full hash。
```

因此，不能简单把 `V3 -> V4` 的巨大提升全部归因于 partial hash。

更严谨的口径是：

```text
评估 partial hash 的真实净收益：
  看 V2 -> V4

评估 V4 修复大文件 batch 回归：
  看 V3 -> V4
```

例如：

```text
large_same_size_different_content:
  V2 76.9 ms -> V4 1.93 ms
  约 39.8x，这是 partial hash 净收益。

large_duplicates:
  V3 516 ms -> V4 78.5 ms
  主要是修复大文件 batch 长尾。

large_duplicates:
  V2 77.4 ms -> V4 78.5 ms
  基本持平，说明真实重复大文件仍然必须 full hash。
```

这个解释非常重要，因为它体现了 benchmark 分析的严谨性。

---

### 6. V5 和 V6 的结论要保持谨慎

V5 引入 `device / inode` 元数据，用于 hardlink 折叠和 inode 排序。这在工程上是合理的，但当前 benchmark 没有专门构造 hardlink-heavy 数据集，也没有在真实磁盘冷缓存环境下验证 inode 排序，因此不能夸大性能收益。

V6 是 buffer 优化的延续，包括 right-sizing 和 `thread_local` scratch buffer。right-sizing 在部分小文件 workload 上有小幅收益，但 `thread_local` 没有证明稳定额外收益。因此，V6 更适合定位为“小文件固定成本的收口优化”，不是新的架构级提速。

## 当前 Benchmark 的边界

当前 benchmark 能支持主要优化结论，但也存在边界。

第一，运行环境提示 CPU scaling enabled，因此小幅波动不应该被解释成确定结论。

第二，数据生成在 `/tmp` 下，容易受到 tmpfs、Page Cache 和热缓存影响。因此，小文件 batch、inode 排序、真实磁盘访问局部性等结论还需要在冷缓存和真实磁盘环境下补充验证。

第三，V6 是当前工作区补充实验，不是历史 commit 序列中的严格 A/B，因此它适合验证 buffer 固定成本是否还有优化空间，但不应把所有混合场景变化都直接归因于 V6。

第四，V5 没有单独 benchmark，因此 hardlink 折叠和 inode 排序目前属于工程上合理、语义上安全，但性能收益仍需专项 workload 验证。

后续可以补充这些 benchmark：

| Benchmark                     | 目的                                           |
| ----------------------------- | -------------------------------------------- |
| `hardlink_heavy`              | 验证 hardlink 折叠能否减少 full hash 次数              |
| `cold_cache_many_small`       | 在真实磁盘冷缓存下验证 inode 排序效果                       |
| `partial_hash_false_positive` | 构造头尾相同但中间不同的大文件，评估 partial hash 假阳性          |
| `large_file_batch_sweep`      | 比较大文件 batch size 为 1 / 2 / 4 / 8 时的负载均衡      |
| `buffer_size_sweep`           | 比较不同 buffer size 对小文件和中等文件的影响                |
| `buffer_reuse_ablation`       | 在同一代码基线上隔离 right-sizing 与 thread_local 的真实收益 |

## 面试讲法

面试时可以把这轮优化讲成一个完整工程闭环：

> 我做 `dedup` 性能优化时，没有一开始就盲目换 hash 算法，而是先分析文件去重的成本模型。不同文件形态瓶颈不一样：小文件主要是每个文件都会发生的固定成本，大文件主要是 full hash 带来的无效 I/O。
>
> 第一阶段我先建立并发执行基线。文件之间 hash 计算互相独立，所以我用自定义 C++11 线程池并发处理文件 hash。benchmark 证明这个方向有效，比如 `1000_files_256KiB` 从 1 线程到 8 线程接近 6.93 倍加速。
>
> 第二阶段我做过滤漏斗。先用 size 分组过滤掉大小唯一的文件，因为大小不同一定不重复；再用 device / inode 识别 hardlink，避免同一物理文件重复 hash。
>
> 第三阶段我按文件大小拆成小文件和大文件两条轨道。小文件直接 full hash，但重点优化每文件固定成本。最典型的是把 `std::vector<char>(bufferSize_)` 改成 `std::unique_ptr<char[]>`，避免每次 hash 前无意义清零。这个改动不改变逻辑，但基础 workload 的 8 线程结果普遍提升约 3 倍。
>
> 第四阶段是大文件优化。大文件 full hash 很贵，所以我加入 head + tail partial hash。对 size 相同但内容不同的大文件，只读头尾几 KiB 就能过滤掉，避免完整读取。相对 V2，非重复大文件场景提升约 40 倍；而真实重复大文件基本不变，因为最终必须 full hash，这也说明优化没有牺牲正确性。
>
> 最后我根据 benchmark 修正策略。比如小文件 batch 在当前环境下没有证明明显收益，但它暴露了大文件不能统一 batch 的边界，所以后续大文件 full hash 使用 Batch Size = 1，避免长尾任务。

更短的版本可以这样说：

> 这个优化的核心不是简单加线程，而是基于文件形态拆瓶颈。线程池解决执行效率，size 和 inode 减少候选，小文件降低每文件固定成本，大文件用 head + tail partial hash 减少无效 I/O，最终 full hash 兜底保证正确性。benchmark 里，V2 的 buffer 清零优化让基础 workload 普遍提升约 3 倍；V4 的 partial hash 让非重复大文件相对 V2 提升约 40 倍；真实重复大文件基本不变，符合必须 full hash 的预期。

## 最终总结

`dedup` 的性能优化最终可以概括为：

```text
基于成本模型设计过滤漏斗，
再用 benchmark 验证每个优化假设。
```

它体现了一个完整的工程优化过程：

```text
发现问题
  -> 建立成本模型
  -> 设计分层优化
  -> 用 benchmark 验证
  -> 根据数据修正策略
```

最终结论是：

```text
小文件优化：
  重点降低每文件固定成本。
  V2 buffer zero-initialization 优化是主要收益来源。

大文件优化：
  重点减少无效 full hash。
  V4 head + tail partial hash 对非重复大文件效果显著。

调度优化：
  小文件可以尝试 batch。
  大文件必须保持细粒度调度，避免长尾。

文件系统优化：
  device / inode 折叠 hardlink 在工程上合理。
  但还需要 hardlink-heavy 和 cold-cache benchmark 进一步验证。
```

这轮优化的价值不只是某个具体技巧，而是形成了一套清晰的方法论：先分析瓶颈，再做针对性优化，最后用数据验证和修正结论。
