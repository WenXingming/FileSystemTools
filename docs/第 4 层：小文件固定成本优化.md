# 第 4 层：小文件固定成本优化

## 优化背景

在前面的流程中，`dedup` 已经完成了几层优化：

```text
并发执行优化：
  使用线程池并发执行 hash 任务，提高 hash 阶段吞吐

第 1 层 size 过滤：
  排除大小唯一、一定不可能重复的文件

第 2 层硬链接折叠：
  避免多个路径指向同一物理文件时重复 I/O 和重复 hash

第 3 层小文件与大文件分流：
  将候选文件拆成 smallCandidates 和 largeCandidates
```

经过第 3 层分流之后，小文件会进入小文件轨道。对于小文件来说，继续做 partial hash 意义不大，因为文件本身就很小，直接 full hash 的读取数据量并不大。但是，这并不意味着小文件没有优化空间。

小文件场景下真正明显的问题，往往不是单个文件的 hash 计算本身，而是每个文件都会重复发生的固定成本。当目录中存在几千甚至几万个小文件时，这些固定成本会被不断放大，最终成为整体耗时中非常重要的一部分。因此，小文件优化的核心思路是：

```text
不重点追求“少读数据”，
而是重点降低“每个文件都会发生的固定成本”。
```

对一个文件计算 full hash，大致流程可以抽象为：

```text
打开文件
  -> 准备 buffer
  -> 读取文件内容
  -> 更新 hash 状态
  -> 关闭文件
  -> 返回 HashResult
```

对于大文件来说，主要成本通常来自读取大量内容和计算完整 hash。

但对于 4KiB 这类小文件来说，文件内容本身很少，真正占比明显的反而可能是：

```text
文件打开 / 关闭
buffer 分配
buffer 初始化
线程池任务提交
future 管理
系统调用固定成本
HashResult 构造与收集
```

这些成本和文件大小关系不大。即使文件只有 4KiB，也仍然要打开文件、准备 buffer、提交任务、等待 future、返回结果。

因此，当文件数量变多时，小文件场景会呈现出一个典型特征：

```text
单个文件很小，
但每个文件都会重复付出一遍固定成本。
```

这就是后续 V2、V3、V6 优化的共同背景。

## V2：避免 buffer zero-initialization

### 问题发现

在 V1 多线程 full hash 基线中，文件 hash 阶段已经可以并发执行。但是，V1 的 `hash_file()` 内部仍然存在一个固定成本问题：每次 hash 文件时都会创建读缓冲区。原来的写法类似于：

```cpp
std::vector<char> buffer(bufferSize_);
```

这个写法看起来很自然，但它有一个隐藏成本：`std::vector<char> buffer(bufferSize_)` 会分配一块 `bufferSize_` 大小的内存，并对其中的元素进行初始化。对于 `char` 来说，这通常意味着整块 buffer 会被清零。

但是在文件读取场景中，这个清零是没有必要的。因为 buffer 创建之后，马上就会被 `read()` 读取到的文件内容覆盖：

```text
创建 buffer
  -> buffer 被清零
  -> read() 读取文件内容
  -> 文件内容覆盖 buffer
```

也就是说，清零这一步并没有参与最终结果，只是在每个文件 hash 前额外做了一次无意义的内存写入。

对于大文件来说，这个成本可能被真实 I/O 掩盖；但对于大量小文件来说，每个文件都会触发一次 buffer 初始化，这个固定成本就会被不断放大。

### 优化方法

V2 的优化方式是：不再使用会默认清零的 `std::vector<char>(bufferSize_)`，而是改成只分配、不做无意义初始化的 buffer。优化前：

```cpp
std::vector<char> buffer(bufferSize_);
```

优化后：

```cpp
std::unique_ptr<char[]> buffer(new char[bufferSize_]);
```

这个改动不改变去重逻辑，也不改变 hash 算法本身。它只是去掉了每次 hash 文件之前的无意义 buffer 清零。可以把这个优化理解为：

```text
原来：
  每个文件 hash 前，都先清零一块 buffer

现在：
  只分配 buffer，等待 read() 覆盖其中内容
```

由于 `read()` 会覆盖实际读取到的区域，所以只要后续 hash 逻辑只处理真实读取到的字节数，就不需要提前清零整块 buffer。

### Benchmark 结果

V2 是小文件优化主线中最稳定、最容易解释的一步。以 8 线程结果为主要口径，V2 相比 V1 的基础 workload 表现如下：

| Workload            |      V1 |       V2 |     提升 |
| ------------------- | ------: | -------: | -------: |
| `100_files_4KiB`    | 2.41 ms | 0.749 ms | 约 3.22x |
| `1000_files_4KiB`   | 22.1 ms |  6.83 ms | 约 3.24x |
| `10000_files_4KiB`  |  214 ms |  70.2 ms | 约 3.05x |
| `1000_files_256KiB` |  123 ms |  32.1 ms | 约 3.83x |

从结果可以看到，基础 workload 的 8 线程耗时普遍下降到原来的三分之一左右。这说明 V2 优化命中了真实瓶颈：当文件很多时，每个文件上的固定成本会被显著放大。即使每次只是少做一次 buffer 清零，全局收益也会非常明显。

> [!tip]
>
> V2 的意义不在于代码改动复杂，而在于它验证了一个关键判断：小文件场景下，性能瓶颈不一定在 hash 算法本身，而可能在每个文件都会重复发生的固定成本。
>
> 这个判断非常重要。因为如果只从算法层面看，文件去重似乎就是“读文件 + 算 hash”。但 benchmark 表明，工程实现中的内存分配、buffer 初始化、任务管理等细节，在大量小文件场景下同样会决定性能。V2 之后，小文件优化的故事就变得清晰了：在双轨并发后，对剩下必须 hash 的小文件，降低每个文件都会发生的固定成本。V2 正是其中最有代表性的优化。

## V6：Buffer right-sizing 与 thread_local 复用

### 优化背景

V6 是 V2 buffer 优化的延续，仍然围绕“小文件固定成本”展开。V2 已经去掉了无意义的 buffer 清零，但后续仍然存在两个细节问题：

```text
问题 1：
  对 4KiB 小文件，仍然可能申请默认 1MiB buffer
  buffer 尺寸明显大于实际读取量

问题 2：
  每次 hash_file() / hash_partial_file() 调用都可能重新分配临时 buffer
```

也就是说，V2 去掉了“清零”成本，但 buffer 的大小和复用方式仍然可以继续优化。因此，V6 拆成两个微步：

1. 微步 1：Buffer right-sizing。根据实际读取需求缩小 buffer 大小
2. 微步 2：thread_local scratch buffer。每个 worker 线程复用自己的临时 buffer

### thread_local buffer

第二个微步是使用 `thread_local` 复用临时 buffer。思路是：每个 worker 线程保留一块自己的 scratch buffer。

```text
每个 worker 线程：
  拥有一个 thread_local buffer
  容量不足时扩容
  容量足够时直接复用
```

这样可以避免每次 `hash_file()` 都重新分配临时 buffer。

**这里没有引入通用内存池**。原因是当前场景并不需要复杂的全局内存管理。文件 hash 只需要每个 worker 线程复用一块读缓冲区，`thread_local` 更简单，也避免了全局内存池可能带来的锁竞争、借还协议和生命周期管理问题。不过，benchmark 没有证明 `thread_local` 带来稳定的额外收益：

| Workload                | V6 right-sizing | V6 thread_local | 结论     |
| ----------------------- | --------------: | --------------: | -------- |
| `1000_files_4KiB`       |         7.41 ms |         7.26 ms | 略有改善 |
| `10000_files_4KiB`      |         65.0 ms |         66.1 ms | 略慢     |
| `many_small_duplicates` |          132 ms |          136 ms | 略慢     |

因此，`thread_local` 的定位应该谨慎：

```text
工程上合理，可以减少未来高并发或更大文件数下的频繁分配；
但在当前 benchmark 中，没有证明它带来稳定的额外加速。
```

### Buffer right-sizing

`Buffer right-sizing` 的核心思路是：不要为小文件申请远大于实际读取量的 buffer。对于 full hash，可以使用：

```text
bufferSize = min(defaultBufferSize, file.size)
```

对于 partial hash，也可以根据实际读取的 head / tail chunk 大小决定 buffer 尺寸。

这样做的好处是：

```text
减少无意义的大块内存分配
减少缓存 / TLB 压力
让小文件 hash 更贴合实际数据大小
```

从 benchmark 看，right-sizing 的方向是合理的：

| Workload            |      V4 | V6 right-sizing | 结论                                     |
| ------------------- | ------: | --------------: | ---------------------------------------- |
| `1000_files_4KiB`   | 11.2 ms |         7.41 ms | 改善明显                                 |
| `10000_files_4KiB`  | 69.8 ms |         65.0 ms | 小幅改善                                 |
| `1000_files_256KiB` | 37.1 ms |         26.9 ms | 有改善，但可能包含当前工作区其他改动影响 |

这些结果说明，小文件固定成本仍然有继续压缩空间。尤其是 `1000_files_4KiB` 从 `11.2 ms` 降到 `7.41 ms`，说明 buffer 尺寸过大确实可能带来额外成本。

## V3：小文件 batch 尝试

### 优化假设

V2 去掉 buffer 清零之后，小文件 hash 阶段的固定成本已经明显下降。此时我继续思考：当 buffer 初始化成本被降低后，新的瓶颈会不会转移到任务调度和 future 管理上？因为在 V1 / V2 的模型中，任务粒度是：

```text
1 个文件 = 1 个线程池任务 = 1 个 future
```

对于几万个小文件来说，这意味着会产生大量线程池任务和大量 future。即使每个文件的实际 hash 很快，任务提交、队列同步、future 管理和结果收集仍然可能形成额外开销。

因此，V3 尝试将小文件做批量切片。旧模型：

```text
1 个文件 = 1 个任务 = 1 个 future
```

新模型：

```text
一批文件 = 1 个任务 = 1 个 future
```

目标是减少任务数量，降低任务队列争用和 future 管理成本。

### 实现思路

V3 的基本思路是把候选文件切成多个 batch：

```text
smallCandidates
  -> create_batches()
  -> batch1, batch2, batch3, ...
```

然后每个 batch 作为一个线程池任务提交：

```text
batch = 1 个线程池任务
```

worker 拿到一个 batch 后，在同一个任务内部顺序处理这一批文件，并返回一组 `HashResult`。

这样可以把原来的大量小任务压缩成较少的批量任务：

```text
原来：
  10000 个文件
  -> 10000 个任务
  -> 10000 个 future

现在：
  10000 个文件
  -> 若干个 batch
  -> 若干个 future
```

从优化动机上看，这个方向是合理的。它试图减少小文件场景下的任务调度成本。

### Benchmark 结果与修正

但是，V3 的 benchmark 结果并没有证明小文件 batch 在当前环境下有明显收益。

以 8 线程结果对比 V2：

| Workload                      |      V2 |      V3 | 结论     |
| ----------------------------- | ------: | ------: | -------- |
| `10000_files_4KiB`            | 70.2 ms | 70.1 ms | 基本持平 |
| `many_small_unique_same_size` |  117 ms |  119 ms | 略慢     |
| `many_small_duplicates`       |  121 ms |  137 ms | 略慢     |

这说明在当前 benchmark 环境下，单文件任务调度并不是最主要瓶颈。由于测试数据在 `/tmp` 下生成，Page Cache 较热，真实磁盘 I/O 和系统调用成本可能被部分掩盖，因此 batch 的收益没有明显体现出来。

更重要的是，V3 暴露了一个策略边界问题：batch 不能无差别地应用到所有文件。如果把同样的 batch 策略用于大文件，就会造成严重的负载不均衡。例如一个 worker 如果拿到 64 个大文件，就会长时间运行；其他 worker 可能很快完成任务并空闲，最终整体耗时被最慢的 batch 拖住。

因此，V3 的价值不是证明“小文件 batch 一定更快”，而是帮助我们明确了一个重要边界：

```text
小文件可以尝试 batch，
但大文件必须保持细粒度调度。
```

这个结论直接影响了后续 V4 的设计：大文件使用 `Batch Size = 1`，避免长尾任务。

## 小文件优化的整体结论

小文件优化这一部分，不能简单讲成“我做了几个小优化”。更好的讲法是：我先建立了小文件成本模型，然后用 benchmark 不断验证和修正假设。整个过程可以概括为：

```text
第一步：发现问题
  大量小文件下，单个文件数据很少，但每个文件固定成本很高

第二步：优化 buffer 清零
  去掉 std::vector<char>(bufferSize_) 带来的无意义初始化
  这是最稳定、收益最明显的一步

第三步：尝试任务粒度优化
  将 1 文件 1 任务改成 batch
  但 benchmark 没有证明小文件明显收益
  同时暴露出大文件不能统一 batch 的边界

第四步：继续收口 buffer 成本
  通过 right-sizing 减少过大 buffer
  通过 thread_local 尝试复用临时缓冲区
  right-sizing 有一定收益，thread_local 没有稳定额外收益
```

这条线的核心不是某个单独技巧，而是：

```text
围绕“小文件固定成本”这个成本模型，
提出假设，
做实现，
用 benchmark 验证，
再根据数据修正策略。
```

## 小结

> [!note]
>
> 叙述时可以这样讲：在做完线程池和 size 过滤之后，我发现小文件场景的瓶颈并不完全在 hash 计算本身。因为 4KiB 这类小文件有效数据很少，但每个文件都要经历 open、read、close、buffer 分配、任务提交和 future 管理。当文件数量达到几千或几万个时，这些每文件固定成本会被放大。
>
> 所以我把小文件优化的目标定义成“降低每文件固定成本”。第一步是优化 buffer 初始化。原来每次 hash 文件时使用 `std::vector<char> buffer(bufferSize_)`，它会分配并清零整块 buffer，但后续 `read()` 会马上覆盖这块内存，所以清零是无意义的。我把它改成 `std::unique_ptr<char[]> buffer(new char[bufferSize_])`，只分配不清零。这个改动不改变去重逻辑，但 benchmark 结果很明显：基础 workload 的 8 线程结果普遍提升约 3 倍，比如 `1000_files_4KiB` 从 `22.1 ms` 降到 `6.83 ms`，`10000_files_4KiB` 从 `214 ms` 降到 `70.2 ms`。
>
> V2 之后，我继续怀疑瓶颈可能转移到任务调度和 future 管理，所以尝试了小文件 batch，把“1 个文件 = 1 个任务 = 1 个 future”改成“一批文件 = 1 个任务 = 1 个 future”。这个方向理论上可以减少任务数量，但 benchmark 没有证明小文件场景明显变快，`10000_files_4KiB` 基本持平，专项小文件 workload 还有轻微退化。更重要的是，它暴露了一个边界：batch 不能直接用于大文件，否则一个 worker 拿到多个大文件会造成长尾，负载均衡会变差。
>
> 后面我又做了 buffer right-sizing 和 `thread_local` scratch buffer。right-sizing 的思路是小文件不要申请远大于实际读取量的 buffer，它在部分小文件 workload 上有改善；`thread_local` 复用工程上合理，但 benchmark 没有证明稳定额外收益。所以这部分最终结论是：V2 是小文件固定成本优化里最稳定的收益来源，V3 是一次任务粒度实验，V6 是固定成本的收口优化。
>
> 这个过程对我比较重要的一点是：我不是盲目加优化，而是先建立成本模型，再用 benchmark 验证。数据支持的优化就保留为主要结论，数据不支持的优化就明确说明边界。

第 4 层小文件优化的核心是降低每文件固定成本。

对于小文件来说，读取字节数并不多，真正容易被放大的成本是文件打开关闭、buffer 分配和初始化、任务提交、future 管理等固定成本。

V2 通过避免 buffer zero-initialization 取得了最稳定的收益，证明小文件固定成本确实是关键问题。V3 尝试通过 batch 减少任务数量，但当前 benchmark 没有证明明显收益，同时暴露了大文件不能统一 batch 的调度边界。V6 继续通过 buffer right-sizing 和 `thread_local` scratch buffer 做收口优化，其中 right-sizing 有一定收益，`thread_local` 没有稳定额外收益。

因此，这一层优化的最终定位是：

```text
V2 是主要收益；
V3 是策略实验和边界识别；
V6 是固定成本收口优化。
```

这一层解决的是小文件轨道的问题。接下来，大文件轨道的主要矛盾会变成：如何减少 size 相同但内容不同的大文件带来的无效 full hash。
