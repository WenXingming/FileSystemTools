# 去重系统哈希算法选择

这篇文档用于说明在文件去重场景里，如何在**正确性、性能、工程复杂度**之间做取舍。

## 一句话结论

当前版本选择 **BLAKE3-256** 作为全量文件哈希。

原因是：它有密码学哈希的低碰撞风险，同时又比传统 SHA-256 更适合高吞吐文件扫描场景。配合文件大小分组、局部哈希过滤和多线程调度，可以在保证误判风险极低的前提下减少不必要的全量读取。

如果是更早期的 MVP 版本，使用过 **FNV-1a 64-bit** 也可以解释得通：它足够简单、零依赖、适合快速验证架构。但它不能作为生产级最终判等依据。

## 去重为什么需要哈希

文件去重不能一开始就逐字节比较所有文件，否则成本太高。更合理的做法是逐层过滤：

1. 先按文件大小分组：大小不同一定不是重复文件。
2. 对大文件做局部哈希：先读头尾少量内容，过滤掉明显不同的文件。
3. 对剩余候选做全量哈希：只有大小相同、全量哈希相同的文件才进入重复组。

所以哈希在这里不是单独存在的算法题，而是整个“漏斗式过滤”流程中的关键一步。

## 为什么第一版可以选择 FNV-1a

第一版使用 FNV-1a 的理由主要是工程取舍。

### 1. 简单、零依赖

FNV-1a 的实现只需要维护一个 64 位状态，循环里做 XOR 和乘法即可。它不依赖 OpenSSL 之类的外部库，适合项目早期快速落地。

### 2. 流式处理很自然

文件可能很大，不能一次性读入内存。FNV-1a 可以边读边更新哈希状态，和分块读取模型天然匹配。

### 3. 早期性能足够好

本地文件去重通常会受到磁盘 I/O 影响。早期选择轻量的非密码学哈希，可以避免在原型阶段把 CPU 计算变成主要瓶颈。

## FNV-1a 的问题

FNV-1a 的问题不是“不能用”，而是**不能直接信任它做最终判等**。

### 1. 64 位输出存在碰撞风险

FNV-1a 64-bit 的输出空间只有 64 位。文件数量越多，两个不同文件得到相同哈希的概率就越高。即使概率不高，只要工具可能据此删除文件，就必须把误判风险当成严重问题处理。

### 2. 它不是抗碰撞哈希

FNV-1a 是非密码学哈希，设计目标是快和简单，不是抵抗构造碰撞。对于普通本地文件，它通常表现不错；但如果把它当成“文件内容唯一身份证”，这个前提就太强了。

### 3. 正确做法是加最终内容确认

如果继续使用 FNV-1a 或 XXH3 这类非密码学哈希，稳妥方案是：

> 大小相同 + 哈希相同，只说明“高度疑似重复”，最后仍要做逐字节比较，才能确认两个文件真的相同。

这样可以保证正确性，但代价是：遇到大量候选文件时，最终比较会带来额外 I/O。

## 为什么当前版本选择 BLAKE3-256

当前版本使用 BLAKE3-256，是为了把“高性能”和“低碰撞风险”同时兼顾起来。

### 1. 碰撞风险极低

BLAKE3 输出 256 位摘要，属于现代密码学哈希。工程上可以把“大小相同 + BLAKE3-256 相同”视为同一内容，碰撞概率远低于 FNV-1a 64-bit。

这里要注意措辞：不能说“100% 不会碰撞”。任何固定长度哈希理论上都有碰撞。更准确的说法是：在正常工程规模和非恶意场景下，BLAKE3-256 的碰撞风险低到可以接受。

### 2. 性能比传统密码学哈希更适合文件扫描

传统印象里，密码学哈希通常比非密码学哈希慢。BLAKE3 的优势在于它是为现代 CPU 设计的，能够利用 SIMD，并且内部结构适合并行计算。

因此，BLAKE3 既比 SHA-256 这类传统方案更适合高吞吐文件读取，又比 FNV-1a 提供更强的可靠性边界。

### 3. 和当前架构匹配

当前去重流程已经有外层并发：多个文件可以交给线程池并行哈希。BLAKE3 的单次哈希接口也适合流式更新，和现有的分块读取逻辑匹配。

面试里可以这样表达：

> 我没有把问题简化成“哪个哈希最快”，而是看整个系统瓶颈。文件去重要先保证不能误判，其次才是吞吐。BLAKE3 是现代密码学哈希，同时性能又足够好，所以它比 FNV-1a 更适合作为最终文件指纹。

## 为什么不直接选 MD5、SHA-1 或 SHA-256

### MD5 / SHA-1

MD5 和 SHA-1 已经不适合作为强调抗碰撞能力的选择。虽然在普通非恶意去重场景里仍可能够用，但面试中选择它们很难解释出优势。

### SHA-256

SHA-256 的可靠性没有问题，但通常性能不如 BLAKE3，也不如 BLAKE3 适合现代并行计算模型。如果目标只是“安全”，SHA-256 可以；如果目标是“安全 + 高吞吐”，BLAKE3 更合适。

## XXH3 是否也是好选择

是的，XXH3 也是很好的工程选择，尤其适合：

- 本地非恶意文件去重；
- 追求极高吞吐；
- 可以接受最后逐字节确认。

如果采用 **XXH3-128 + 最终逐字节比较**，也是非常稳妥的方案。它的思路是用极快的非密码学哈希做分桶，再用真实内容比较保证正确性。

与 BLAKE3 相比：

- XXH3 更偏性能分桶；
- BLAKE3 更偏内容指纹；
- 如果系统可能跨机器缓存哈希、长期保存哈希、或者希望减少最终二次读取，BLAKE3 更容易解释。

## 面试回答模板

可以这样回答：

> 这个项目里我最终选择了 BLAKE3-256。文件去重的核心风险是误判重复文件，所以全量哈希不能只看速度，还要考虑碰撞风险。
>
> 第一版我曾考虑过 FNV-1a 64-bit，因为它实现简单、零依赖、可以流式更新，适合 MVP 阶段验证“按大小分组、局部哈希、全量哈希”的整体流程。但 FNV-1a 是非密码学 64 位哈希，不能直接作为最终判等依据。如果继续用它，就必须在哈希相同后做逐字节确认。
>
> 后来我选择 BLAKE3，是因为它在可靠性和性能之间比较平衡。它输出 256 位摘要，碰撞风险在工程上非常低；同时它比传统 SHA-256 更适合现代 CPU 和高吞吐文件扫描。结合文件大小分组和局部哈希过滤，可以减少全量读取的范围，又能让最终结果有比较强的可信度。

如果面试官继续追问“为什么不用最快的哈希”，可以补一句：

> 对去重工具来说，最快不是唯一目标。哈希再快，如果误判导致删错文件，系统就是不可接受的。所以我的选择逻辑是：先保证正确性边界，再在这个边界内优化吞吐。

## 总结

- FNV-1a：适合原型，优点是简单、零依赖、流式友好；缺点是 64 位非密码学哈希，不能直接信任。
- XXH3-128：适合高性能分桶，但建议保留最终逐字节确认。
- BLAKE3-256：适合作为当前版本的文件内容指纹，兼顾低碰撞风险和高吞吐。

最终选择 BLAKE3，不是因为它“绝对不会碰撞”，而是因为它在这个项目的正确性要求、性能目标和工程复杂度之间，是更均衡的选择。
