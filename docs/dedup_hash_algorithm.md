# 去重系统哈希算法选择

在文件去重场景里，哈希算法不是单纯比较“谁最快”，而是在**正确（碰撞）性、性能、工程复杂度**之间做取舍。

当前版本选择 **BLAKE3-256** 作为全量文件哈希。它的优势是：有现代密码学哈希的低碰撞风险，同时又比传统 SHA-256 更适合高吞吐文件扫描。结合文件大小分组、局部哈希过滤和多线程调度，可以先尽量缩小候选范围，再对少量候选计算完整内容指纹。

更早期的 MVP 版本选择 **FNV-1a 64-bit** 也能解释得通：它简单、零依赖、流式友好，适合快速验证去重流程。但它不能作为生产级最终判等依据。

## 去重流程里的哈希角色

文件去重不能一开始就把所有文件两两逐字节比较，那样 I/O 成本太高。更合理的做法是漏斗式过滤：

1. 先按文件大小分组：大小不同一定不是重复文件。
2. 对大文件做局部哈希：读取头尾少量内容，过滤掉明显不同的文件。
3. 对剩余候选做全量哈希：大小相同、全量哈希相同的文件才进入重复组。

所以哈希在这里承担的是“快速缩小候选集合”的角色。解释时可以强调：这个选择不是孤立的算法选择，而是服务于整个去重流水线。

## FNV-1a：为什么适合原型，不适合最终判等

FNV-1a 是一个非常轻量的非密码学哈希。它的核心更新逻辑可以理解为：

```text
hash = hash xor current_byte
hash = hash * FNV_prime
```

也就是每读入一个字节，就把这个字节混入当前 64 位状态，再乘以一个固定质数。它只需要维护一个整数状态，不依赖外部库，可以很自然地支持分块读取：文件读一块，就更新一次哈希状态。

这就是它适合 MVP 阶段的原因。文件去重早期最重要的是先验证流程是否成立：能不能遍历文件、按大小分组、做局部哈希、再做全量哈希。FNV-1a 代码少、接入快、性能开销低，不会在原型阶段引入复杂依赖。

但 FNV-1a 的边界也要讲清楚。它是 64 位非密码学哈希，设计目标是简单和快速，不是抵抗碰撞。文件数量变多时，64 位输出空间会带来生日悖论下的碰撞风险；如果存在恶意输入，它也不能提供密码学意义上的抗碰撞保证。

另外，FNV-1a 的计算是强串行的：下一个字节的计算依赖上一个字节算出的 `hash`。这意味着它很难充分利用现代 CPU 的 SIMD 并行能力。它快，主要是因为逻辑简单；但它不是现代高吞吐哈希里最能压榨硬件的方案。

因此，如果继续使用 FNV-1a，正确做法是把它当成“候选分桶”工具：

> 大小相同 + FNV-1a 哈希相同，只能说明两个文件高度疑似重复，最后仍然需要逐字节比较确认。

这样可以保证不误判，但代价是候选文件多时会产生额外二次读取。

## BLAKE3：为什么当前版本选择它

BLAKE3 是现代密码学哈希，当前代码使用 256 位输出作为文件内容指纹。这里要注意措辞：任何固定长度哈希理论上都有碰撞，不能说“100% 不会碰撞”。更准确的说法是：在正常工程规模和非恶意文件去重场景下，BLAKE3-256 的碰撞风险低到可以接受，远低于 FNV-1a 64-bit。

BLAKE3 的关键不只是“更安全”，还在于它把密码学哈希做得很快。传统 SHA-256 更偏线性处理：前一个块的状态会影响后一个块，天然不容易大规模并行。BLAKE3 则采用树形结构，可以把输入拆成许多小块，先分别计算块哈希，再像 Merkle Tree 一样逐层合并成根哈希。

这里的 SIMD 可以简单理解成“单条指令处理多份数据”。普通代码可能像排队结账：一次处理一个数据；SIMD 更像同时打开多条收银通道：CPU 的一条向量指令可以同时对多组字节做同一种运算。它和多线程不是一回事：多线程是多个 CPU 核心或多个执行流同时干活，SIMD 是单个核心内部用更宽的向量寄存器一次处理更多数据。

举个直观例子：如果哈希算法要对 8 组数据块做相同的混合运算，普通标量代码可能要按顺序算 8 次；支持 SIMD 的实现可以把这 8 组数据装进向量寄存器里，用一批向量指令一起算。AVX2、AVX-512、NEON 就是不同平台上的 SIMD 指令集名称。

这种结构带来了两个性能优势：

- **SIMD 友好**：同一批数据块可以用 AVX2、AVX-512、NEON 等向量指令同时处理，单个 CPU 核心也能一次处理多路数据。
- **多核友好**：不同数据块之间可以并行计算，理论上可以分配给多个 CPU 核心，再合并中间结果。

这是 BLAKE3 性能高的关键：它不是简单地“加密算法优化得更快”，而是算法结构本身就适合现代处理器的并行能力。

不过结合当前项目，还要补一句工程判断：我们的 `DuplicateFinder` 外层已经用线程池并行处理多个文件，因此不一定需要在单个文件内部再开多线程哈希。否则可能出现线程过多、调度成本上升的问题。更合理的理解是：

> 外层用线程池并行处理多个文件；内层用 BLAKE3 的流式接口和 SIMD 能力高效处理每个文件的数据块。

这样既匹配现有架构，也避免把“BLAKE3 支持多核并行”误讲成“当前实现一定对单个文件启用了内部多线程”。

## 为什么不是 MD5、SHA-1、SHA-256 或 XXH3

MD5 和 SHA-1 已经不适合作为强调抗碰撞能力的选择。虽然普通非恶意本地去重里它们未必马上出问题，但选择它们很难解释出优势。

SHA-256 的可靠性没有问题，但通常不如 BLAKE3 适合高吞吐文件扫描。它更保守、更通用；如果目标是“安全 + 高吞吐”，BLAKE3 更容易解释。

XXH3 也是一个很好的工程选择，尤其适合本地非恶意去重。它非常快，`XXH3-128 + 最终逐字节比较` 是稳妥方案：用非密码学哈希快速分桶，再用真实内容比较保证正确性。它和 BLAKE3 的区别是：XXH3 更像高性能分桶工具，BLAKE3 更适合作为长期保存、跨机器比对或缓存使用的内容指纹。

## 回答模板

可以这样回答：

> 这个项目里我最终选择了 BLAKE3-256。文件去重最重要的风险是误判重复文件，所以全量哈希不能只看速度，还要考虑碰撞风险。
>
> 第一版我可以用 FNV-1a 64-bit，因为它实现简单、零依赖、可以流式更新，很适合 MVP 阶段验证“按大小分组、局部哈希、全量哈希”的整体流程。但 FNV-1a 是非密码学 64 位哈希，不能直接作为最终判等依据。如果继续用它，就必须在哈希相同后做逐字节确认。
>
> 后来我选择 BLAKE3，是因为它在可靠性和性能之间比较平衡。它输出 256 位摘要，工程上碰撞风险极低；同时它内部是树形结构，天然适合 SIMD 和多核并行，比传统 SHA-256 更适合高吞吐文件扫描。结合文件大小分组和局部哈希过滤，可以减少全量读取范围，又能让最终结果有比较强的可信度。

如果追问“为什么不用最快的哈希”，可以补一句：

> 对去重工具来说，最快不是唯一目标。哈希再快，如果误判导致删错文件，系统就是不可接受的。所以我的选择逻辑是：先保证正确性边界，再在这个边界内优化吞吐。

总结一下：FNV-1a 适合原型，XXH3 适合高性能分桶，BLAKE3-256 更适合作为当前版本的最终文件内容指纹。最终选择 BLAKE3，不是因为它“绝对不会碰撞”，而是因为它在正确性要求、性能目标和工程复杂度之间更均衡。
