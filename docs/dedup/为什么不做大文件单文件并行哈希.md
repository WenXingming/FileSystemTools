# dedup 大文件单文件并行哈希评估

## 核心结论

这个优化思路在理论上是成立的，但不能简单理解为“把文件切成几块，各线程分别计算 BLAKE3，最后把几个 hash 合并起来”。BLAKE3 确实是一种树形哈希算法，天然具备并行计算不同子树的能力；但要得到与当前顺序实现完全相同的标准 BLAKE3 摘要，线程之间需要合并的是带有正确位置和树结构信息的内部状态，而不是每一块已经 `finalize` 的最终摘要。

结合当前项目，文件内并行最适合解决的是“参与全量哈希的超大候选文件很少，无法占满线程池”的问题。例如，经过文件大小和首尾局部哈希过滤后，只剩两个 50 GiB 文件需要完整计算，而线程池有八个 worker。当前实现只能让两个线程分别处理一个文件，其余线程处于空闲状态；如果能够把每个文件再拆成多个 BLAKE3 子树，就有机会让更多 CPU 核心参与，缩短这两个文件的处理时间。

如果目录中有足够多的候选文件，当前的文件级并行通常已经是更简单、更稳妥的方案。现有代码把每个大文件作为独立任务提交给线程池，多个文件可以同时进行顺序读取和哈希计算。此时再为每个文件增加内部并行，不但不一定提高总吞吐，还可能造成线程争用、内存带宽竞争和更混乱的磁盘访问。因此，单文件并行应该被定位为文件级并行的补充策略，而不是替代策略。

综合判断是：值得继续做专项 benchmark 和最小原型，但暂时不建议直接修改生产哈希路径。是否采用这项优化，应由“少量超大候选文件”场景中的实际墙钟收益决定，而不是仅凭 BLAKE3 支持树形计算这一点决定。

## 为什么理论可行，实际收益却不确定

当前 `Hasher::hash_file` 使用一个线程顺序读取文件，每次最多读取 1 MiB，再调用 BLAKE3 的流式 `update`。外层 `DuplicateFinder` 使用线程池并发处理不同文件，并且进入全量哈希阶段的大文件采用 Batch Size = 1，所以任务调度粒度已经细化到了单个大文件。换句话说，当前结构是“文件之间多线程，单个文件内部单线程”。

BLAKE3 的优势是内部使用 Merkle Tree 式的树形结构。一个大文件可以被划分为多个子树，各子树独立计算 chaining value，最后按照原始位置和规定的树结构逐层归并。这种结构从算法上消除了传统线性哈希必须从头计算到尾的状态依赖，因此多核并行是可行的。

但项目当前依赖的是 BLAKE3 1.5.1 C API，公开接口只有初始化、流式更新和最终输出，没有公开的“计算指定子树并合并 chaining value”接口。现有 C 实现会在单线程内部利用 AVX2、AVX-512 等 SIMD 指令同时压缩多个 chunk，所以当前基线已经不是低效的逐块标量计算。文件内多线程带来的收益，是在现有 SIMD 基础上进一步利用多个 CPU 核心，实际提升不会等于线程数。

最终性能还取决于瓶颈是在 CPU 还是 I/O。不同场景的预期大致如下：

| 场景 | 文件内并行的预期 |
| --- | --- |
| 多个大文件，数量不少于 worker 数 | 文件级并行已经能占满线程池，额外拆分通常收益很小 |
| 少量超大文件，数据位于页缓存或 RAM Disk | 哈希计算更容易成为瓶颈，文件内并行最可能有效 |
| 少量超大文件，位于高速 NVMe SSD | 可能通过多核计算和更高 I/O 队列深度获得收益，需要实测 |
| SATA SSD | 单线程可能已经能跟上存储吞吐，也可能受页缓存影响获益，不能只凭介质名称判断 |
| HDD | 多线程读取相距很远的文件区间可能造成频繁寻道，直接并发 `pread` 存在明显退化风险 |
| NAS、RAID、虚拟磁盘 | 延迟、带宽和缓存策略差异很大，只能通过目标环境测试判断 |

这里需要区分“并行读取”和“并行计算”。在 HDD 上，如果多个线程分别读取文件头部、中部和尾部，原本连续的顺序 I/O 可能变成随机寻道，这是典型的负优化；但如果仍由一个读取线程保持顺序读取，只把已经读入内存的缓冲区交给多个线程计算，则不会破坏磁盘顺序性。后者的代价是需要额外的缓冲区池、任务背压和结果归并。因此，HDD 并不意味着算法并行绝对不可用，而是不能采用粗暴的多区间并发读取。

现有 benchmark 能证明文件级并行有效，但不能回答文件内并行是否有效。当前 `large_duplicates` 使用 20 个 8 MiB 文件，候选数量足以填满线程池。2026-06-27 在当前机器上重新运行时，墙钟均值约为：1 线程 128 ms、2 线程 63.8 ms、4 线程 34.4 ms、8 线程 23.6 ms，1 到 8 线程获得约 5.4 倍加速。这说明候选文件较多时，现有架构已经具有良好的并发扩展性。

这组数据不能被解释为磁盘吞吐，也不能用于判断少量超大文件。benchmark 文件生成后立即被读取，数据通常仍在操作系统页缓存中；单文件只有 8 MiB；测试覆盖的是整个 DuplicateFinder 流程，而不是隔离的 Hasher；此外，当前 Google Benchmark 没有使用 real-time 作为速率统计基准，worker 的 CPU 时间没有被正确计入速率分母，因此输出中高达 80–158 GiB/s 的 `bytes_per_second` 并不是真实存储吞吐。墙钟时间可以用于比较同一 workload 的趋势，但现有吞吐数字不能作为文件内并行的实施依据。

## 实现时真正需要解决的问题

第一个问题是保持摘要兼容。BLAKE3 的基础 chunk 是 1024 字节，但正确分块不只是把边界对齐到 1024 字节。每个子树还需要正确的全局 chunk counter，完整子树通常需要满足 2 的幂次大小和位置对齐，最后一个不完整 chunk 需要特殊处理，父节点和根节点也使用不同 flags。线程计算完成后需要按原始顺序归并内部 chaining value，并在最后一步按根节点规则输出摘要。

因此，下面这种做法是不正确的：

```text
thread 1: BLAKE3(file[0..N])
thread 2: BLAKE3(file[N..2N])
result:   BLAKE3(hash1 || hash2)
```

它得到的是一种自定义分块摘要，并不等于 `BLAKE3(whole_file)`。如果分块大小跟随线程数变化，同一个文件还可能在不同线程配置下得到不同结果。生产实现应使用能够稳定提供 BLAKE3 子树计算和归并能力的后端，或者维护一个经过官方测试向量完整验证的适配层，不能直接依赖 BLAKE3 C 库的私有结构，也不应自行定义新的摘要算法。

第二个问题是线程池调度。不能在当前的大文件 worker 中继续向同一个线程池提交分块任务，然后同步等待这些任务完成。假设八个大文件任务已经占满八个 worker，每个 worker 又提交子任务并等待，所有 worker 都会停在等待状态，而队列中的子任务无人执行，最终形成线程池饥饿甚至死锁。正确方向是在全量哈希阶段开始前，由顶层调度器根据候选文件数量、文件大小和总线程预算统一决定任务粒度，让文件任务和子树任务都成为可以直接执行的叶子任务，再由协调者完成归并。

第三个问题是读取方式和资源上限。多个任务可以使用 `pread` 从同一个文件描述符读取不同区间，避免共享 `ifstream` 文件位置，但这种方式更适合 SSD 或热缓存场景；若要兼顾 HDD，更稳妥的是保留顺序读取并把缓冲区交给计算任务。无论采用哪种方式，都必须限制“并行 lane 数 × 每个 lane 的 buffer 大小”，防止同时处理多个大文件时放大内存占用和文件描述符数量。

并行读取还会放大文件在扫描期间被修改的问题。任务必须正确处理短读、文件截断、读取失败以及哈希前后文件大小或元数据变化，避免不同线程读到文件的不同时刻后仍然生成一个看似有效的混合摘要。这个问题在现有顺序实现中已经存在，但并行区间读取会让它更容易发生，因此实现时应同时补齐相应错误检查。

## 建议的验证和决策路径

第一步应只补 benchmark，不改变生产实现。需要新增两个层次的测试：一类只测 `Hasher`，隔离文件读取与 BLAKE3 计算；另一类测完整去重流程，观察调度和局部哈希漏斗加入后的真实用户耗时。目标 workload 应包含两个真重复的大文件，大小至少覆盖 256 MiB、1 GiB 和 4 GiB，并比较 1、2、4、8 个计算线程。同时保留当前 20 个 8 MiB 及更多文件的 workload，防止新策略优化了少量超大文件，却破坏了已有的文件级并行。

测试结果必须明确区分 warm-cache 与真实存储读取。页缓存场景可以帮助判断 CPU 哈希上限；SSD 和 HDD 场景用于判断端到端收益及随机 I/O 风险。benchmark 应使用墙钟时间计算实际读取吞吐，并记录真实 full-hash 字节数，不能继续用逻辑文件总大小或主线程 CPU 时间推导吞吐。

第二步才制作最小并行原型，而且首先验证正确性。空文件、1 字节、1023/1024/1025 字节、子树边界前后、非 2 的幂次长度和数 GiB 文件都应覆盖；同一文件在不同线程数、不同 buffer 大小下的摘要必须与当前顺序实现完全一致。只有兼容性测试通过后，性能数据才有意义。

如果原型确实有效，生产策略应保持自适应但简单：当全量哈希候选数量足以占满线程池时，继续采用当前文件级并行；只有候选数量明显少于 worker 数，并且文件超过实测有效的大小阈值时，才为单个文件分配多个计算 lane。初期可以提供 `--intra-file-threads` 参数并默认设为 1，让用户显式选择，而不是通过 `--assume-nvme` 之类的参数猜测存储环境。

建议把合入标准固定为：所有线程数和分块方式都生成完全相同的标准 BLAKE3 摘要；目标“少量超大候选文件”场景的墙钟时间稳定改善至少约 15%；现有多文件 workload 没有明显退化；HDD 和未知介质默认仍走保守路径；内存、任务数和文件描述符数量保持有界。如果最终只在热缓存或 RAM Disk 上出现很小收益，而真实去重流程没有稳定改善，就应继续保留当前实现。

最终评价是：这个方向具有明确的理论基础，也确实覆盖了当前架构中的一个并发空洞，但适用范围比“所有大文件”窄得多。它真正服务的是“少量超大碰撞候选”，而不是常规目录扫描。现阶段最有价值的下一步不是立即编写复杂的 BLAKE3 子树归并代码，而是建立能够准确回答收益问题的专项 benchmark。
