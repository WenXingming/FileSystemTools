# 并发执行优化：线程池

串行 full hash → 线程池并发 full hash。

## 优化思路

并发执行是我首先考虑到的一个重要性能优化方向。因为考虑到文件去重任务有一个天然特点：**不同文件之间的处理过程基本没有计算依赖**。计算 `fileA` 的 hash 不需要等待 `fileB` 的 hash 结果，读取 `fileA` 的内容也不会依赖 `fileC` 的内容。因此，文件 hash 阶段非常适合并发处理。

文件 hash 过程通常可以拆成两部分：

```text
文件读取 I/O + hash 计算
```

最朴素的执行方式是单线程逐个处理文件。单线程执行时，文件读取 I/O 等待和 CPU 上 Hash 计算都只能串行推进。这种方式实现简单，但所有文件的读取和 Hash 计算都被串行化。对于大量文件或较大的文件集合来说，整体耗时会随着文件数量和文件体积明显增长。

因此，一个直接的优化思路是：将多个文件的读取和 hash 计算并行执行，让多个 Worker Thread 同时处理不同文件，从而重叠 I/O 等待和 CPU 计算。

> [!tip]
>
> 线程池优化的是执行方式，而不是后续的过滤逻辑。它解决的是“如何更快地执行 Hash”，而不是“如何减少需要 Hash 的文件”。后续的 size 过滤、hardlink 折叠、partial hash 等，解决的才是减少工作量的问题。

## 使用线程池

实现并发最直接的方式是为每个任务创建一个线程，但这种方式不适合文件去重场景。原因是文件数量可能很多。如果每处理一个文件就创建一个线程，**线程的创建和销毁本身会带来明显开销**，尤其在小文件场景下，单个文件的处理时间很短，线程创建销毁的成本甚至可能接近或超过实际任务成本。同时，在这种实现方式下我们很难显式**控制线程并发数量**，大量线程被创建后会造成**调度压力、上下文切换开销和资源占用问题，极端情况下甚至可能触发 OOM**。

因此，我没有采用“每个文件创建一个线程”的方式，而是使用自己实现的 C++11 线程池。我实现的线程池基于 C++11 的 `std::packaged_task` 等机制封装任务，并通过 `auto` 和 `decltype` 实现提交任务时的返回值后置和返回值类型推导，因此支持有返回值的异步任务的提交。调用方提交任务后可以得到一个 `std::future<T>`，之后通过 `future.get()` 获取结果。这符合文件 Hash 任务需要返回 `HashResult` 的需求。

> [!tip]
>
> 我实现的线程池还支持优先级调度和可选自动扩缩容。优先级调度可以让更重要的任务优先执行；自动扩缩容可以根据队列压力和 worker 空闲情况调整线程数量。但在当前项目中暂时还未使用到这些特性。

**整体模型可以抽象为**：

1. 创建固定数量的 Worker Thread 初始化构建线程池；
2. 主线程提交文件 Hash 任务，任务进入线程池任务队列；
3. 线程池中的 Worker Thread 从队列中取任务执行，任务完成后通过 future 返回结果；
4. 主线程统一收集执行结果 HashResult。

在 `dedup` 中引入线程池，不只是简单地“开多个线程”，而是把文件 hash 阶段抽象成可提交、可调度、可返回结果的异步任务系统。

## 并发优化的实现

我实现的线程池支持有返回值任务的提交。在此基础上，文件去重的 hash 阶段可以改造成并发执行。每个文件的 hash 计算被封装成一个独立任务：

```text
1 个文件 = 1 个 hash 任务 = 1 个 future
```

主线程负责遍历文件列表，将任务提交到线程池；线程池中的 Worker Thread 负责读取文件内容并计算 Hash；最后主线程收集所有 `HashResult`，再进行分组。伪代码如下：

```cpp
std::vector<HashResult> DuplicateFinder::compute_hashes(
    const std::vector<FileInfo>& files
) const {
    std::vector<std::future<HashResult>> futures;
    futures.reserve(files.size());

    for (const FileInfo& file : files) {
        futures.emplace_back(
            threadPool_.submit_task([this, file]() {
                return hash_file(file);
            })
        );
    }

    std::vector<HashResult> results;
    results.reserve(files.size());

    for (auto& future : futures) {
        results.emplace_back(future.get());
    }

    return results;
}
```

整体流程可以抽象为：

```text
扫描目录
  -> 主线程收集文件信息
  -> 主线程提交多个 hash 任务到线程池
  -> 线程池 Worker Thread 并发读取文件并计算 hash
  -> 主线程收集所有 HashResult
  -> 按 hash 分组
  -> 输出重复文件组
```

这种单文件任务粒度的优点是：负载均衡比较自然。每个 Worker Thread 一次只拿一个文件任务，执行完再取下一个任务，不容易因为某个 worker 一次拿到过多大文件而长期占用线程。当然，它的代价是任务数量会比较多。大量小文件场景下，`1 个文件 = 1 个任务 = 1 个 future` 会放大任务提交、队列同步和 future 管理成本。这也为后续“小文件 batch”优化埋下了问题背景。

## Benchmark 结果与分析

我使用 Google Benchmark 框架进行了性能基准测试，重点关注 `Time` 指标，因为它代表用户真实等待时间（指标 `CPU` 则关注 `CPU` 执行时间）。Benchmark 结果显示，多线程并发相比单线程有明显收益。

| Workload            | 1 线程  | 2 线程  | 4 线程  | 8 线程  | 1 -> 8 加速 |
| ------------------- | ------- | ------- | ------- | ------- | ----------- |
| `100_files_4KiB`    | 9.92 ms | 6.27 ms | 3.65 ms | 2.41 ms | 4.12x       |
| `1000_files_4KiB`   | 121 ms  | 65.3 ms | 38.1 ms | 22.1 ms | 5.48x       |
| `10000_files_4KiB`  | 1042 ms | 597 ms  | 355 ms  | 214 ms  | 4.87x       |
| `1000_files_256KiB` | 852 ms  | 437 ms  | 227 ms  | 123 ms  | 6.93x       |

从结果可以看到，随着线程数从 1 提升到 8，所有 workload 的耗时都明显下降。这说明文件去重中的 full hash 阶段确实存在可并行化空间。在不同的数据集场景下，加速比有略微差异，但共同点是加速比都具备理想的数值，这说明了说明线程池 full hash 的核心价值：**它可以重叠 I/O 等待和 CPU 计算，从而显著降低 wall time**。

> [!important]
>
> 虽然线程池带来了明显收益，但从 1 线程到 8 线程并没有在所有场景下都达到 8x 线性加速。这是符合预期的。
>
> 主要原因包括：
>
> 1. 任务提交、队列同步和 future 管理本身有开销。
> 2. 文件打开、读取、关闭等系统调用不能完全线性扩展。
> 3. 存储设备吞吐、Page Cache 和操作系统调度都会影响最终结果。
> 4. 小文件任务粒度太细时，固定成本会占据更高比例。
> 5. benchmark 本身也可能受到 CPU scaling 和系统负载影响。
>
> 因此，这里的结论不应该是“线程越多越好”，而应该是：在当前测试环境和这些 workload 下，从 1 线程增加到 8 线程可以显著降低 wall time，说明并发执行方向是有效的。

## 测试环境与适用边界

当前 benchmark 能证明：在当前测试环境（SSD）下，多线程 full hash 相比单线程有明显收益。但这个结论不能无条件外推到所有存储设备和所有运行环境。

对于 SSD 或 NVMe 这类随机访问能力较强的存储设备，多线程读取通常更容易发挥设备并发能力。但如果硬件存储设备是 HDD，过高并发可能导致频繁寻道，破坏顺序读模式，甚至导致磁头抖动，反而降低吞吐。因此多线程并发需要考虑硬件环境，当在 HDD 上时需要谨慎启用。

最后，线程池只能提升执行效率，不能减少需要执行的工作量。如果所有文件都直接 full hash，那么：

```text
小文件场景：
  每个文件的固定成本仍然会被大量放大

大文件场景：
  即使文件最终并不重复，也仍然可能被完整读取和 hash

文件系统层：
  多个路径指向同一物理文件时，仍然可能重复读取同一份数据
```

因此，线程池 full hash 是一个有效的并发执行基线，但不是最终优化方案。后续优化需要继续思考：哪些文件不需要 hash？哪些大文件不需要 full hash？哪些文件其实指向同一份物理数据？

## 小结

并发执行优化是 `dedup` 中首先考虑的性能优化方向。由于不同文件之间没有计算依赖，可以使用线程池将多个文件的读取和 hash 计算并行执行。

我没有采用“每个文件创建一个线程”的方式，而是使用自定义实现的 C++11 线程池。这个线程池通过复用 worker 线程避免频繁创建和销毁线程，并通过 `std::future` 支持有返回值任务的异步提交，也为异常传播、优先级调度和自动扩缩容等能力提供了基础。

Benchmark 结果证明，多线程 full hash 相比单线程有明显收益，尤其在中等文件和大量文件场景下可以显著降低用户等待时间。

但线程池并不改变去重漏斗本身。它解决的是“如何更快地执行 hash”，而不是“如何减少需要 hash 的文件”。因此，在建立并发 full hash 基线之后，后续优化还需要继续围绕过滤和成本模型展开，例如按文件大小过滤、优化小文件固定成本、对大文件引入 partial hash，以及利用 `device / inode` 减少文件系统层面的重复工作。
