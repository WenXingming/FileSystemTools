# 第 3 层：小文件与大文件双轨分流

## 优化背景

经过前两层优化之后，`dedup` 已经完成了两类候选缩减：

```text
第 1 层：按 size 分组
  -> 排除大小唯一、一定不可能重复的文件

第 2 层：硬链接折叠
  -> 折叠多个路径指向同一物理文件造成的重复 I/O
```

此时，剩下的候选文件已经不再是原始目录中的全部文件，而是经过初步过滤和物理折叠后的唯一物理候选文件。

但是，这些候选文件的形态仍然可能差异很大。有些是几 KiB 的小文件，有些是几 MiB 甚至更大的大文件。如果对它们继续使用完全相同的处理策略，就会出现策略不匹配的问题。

小文件和大文件的性能瓶颈并不相同。

对于小文件来说，文件内容本身很少，真正明显的成本往往不是 hash 计算，而是每个文件都会发生的固定成本，例如文件打开和关闭、buffer 分配、任务提交、future 管理等。

对于大文件来说，单个文件的数据量很大。如果仅仅因为 size 相同就直接 full hash，那么一旦这些大文件内容并不相同，就会产生大量无效 I/O 和无效 hash 计算。

因此，在进入后续 hash 阶段之前，需要先根据文件大小对候选文件进行分流。

---

## 核心思路

这一层的核心思路是：**不同大小的文件走不同的处理轨道**。

```text
唯一物理候选文件
  -> 按阈值分流
  -> 小文件轨道
  -> 大文件轨道
```

在当前实现中，可以使用一个阈值将候选文件划分为两类：

```text
file.size <= threshold:
  进入小文件轨道

file.size > threshold:
  进入大文件轨道
```

代码中使用的阈值是 `4096` 字节，也就是 4KiB：

```cpp
std::vector<FileInfo> smallCandidates;
std::vector<FileInfo> largeCandidates;

split_candidates_by_threshold(
    uniquePhysicalFiles,
    4096,
    smallCandidates,
    largeCandidates
);
```

需要注意，这个阈值不是判断文件是否重复的语义条件，而是一个性能策略边界。它的作用是把候选文件交给不同的优化策略处理。

---

## 为什么需要分流

文件去重的成本模型不是单一的。不同文件形态下，主要瓶颈不同。

### 小文件的主要问题：固定成本占比高

对于小文件来说，单个文件的数据量很小，例如 4KiB 文件只需要读取很少的数据就能完成 hash。

但是，每处理一个文件，仍然会发生一些固定成本：

```text
打开文件
  -> 分配或准备 buffer
  -> 读取文件
  -> 更新 hash
  -> 关闭文件
  -> 返回 HashResult
```

这些成本中，很多并不随文件大小线性增长。即使文件只有 4KiB，也需要打开文件、提交任务、管理 future、准备 buffer。

当文件数量达到几千或几万个时，这些“每个文件都会发生一次”的固定成本会被不断放大。

因此，小文件轨道的优化重点不是减少读取字节数，而是降低每个文件的固定开销。

后续小文件优化会重点关注：

```text
buffer 分配与初始化
任务提交与 future 管理
小文件 batch
thread_local scratch buffer
```

---

### 大文件的主要问题：无效 I/O 成本高

对于大文件来说，情况完全不同。

大文件的单个文件体积很大，full hash 意味着必须从头到尾读取整个文件。如果多个大文件只是大小相同，但内容并不相同，那么直接 full hash 就会产生大量无效读取。

例如：

```text
fileA: 8MiB
fileB: 8MiB
fileC: 8MiB
```

它们 size 相同，所以会通过第 1 层 size 过滤。但 size 相同并不代表内容相同。如果直接对它们全部 full hash，就需要读取大量最终没有必要读取的数据。

因此，大文件轨道的优化重点不是减少固定成本，而是减少不必要的完整文件读取。

后续大文件优化会采用 partial hash：

```text
大文件
  -> 先读取头部和尾部少量数据
  -> 计算 partial hash
  -> partial hash 不同，直接跳过 full hash
  -> partial hash 仍然碰撞，才进入 full hash
```

这样可以把昂贵的 full hash 留给真正需要进一步验证的大文件候选。

---

## 实现方式

候选文件分流可以封装成一个独立函数：

```cpp
void split_candidates_by_threshold(
    const std::vector<FileInfo>& files,
    uint64_t threshold,
    std::vector<FileInfo>& smallCandidates,
    std::vector<FileInfo>& largeCandidates
) {
    for (const FileInfo& file : files) {
        if (file.size <= threshold) {
            smallCandidates.push_back(file);
        } else {
            largeCandidates.push_back(file);
        }
    }
}
```

在主流程中，它位于 hardlink 折叠之后：

```cpp
std::vector<FileInfo> uniquePhysicalFiles;
std::map<std::pair<uint64_t, uint64_t>, std::vector<FileInfo>> hardlinkFollowers;

// hardlink 折叠后得到 uniquePhysicalFiles

std::vector<FileInfo> smallCandidates;
std::vector<FileInfo> largeCandidates;

split_candidates_by_threshold(
    uniquePhysicalFiles,
    4096,
    smallCandidates,
    largeCandidates
);
```

经过这一步之后，原本统一的候选文件集合被拆成两条轨道：

```text
uniquePhysicalFiles
  -> smallCandidates
  -> largeCandidates
```

后续就可以分别选择更适合的处理策略。

---

## 小文件轨道

小文件轨道中的文件体积较小，继续做 partial hash 意义不大。

原因是：小文件本身就只有很少的数据。如果为了避免 full hash 再额外设计 partial hash，可能反而增加额外判断和调度开销。对于这类文件，更直接的策略是进入 full hash。

流程可以抽象为：

```text
smallCandidates
  -> 按 device / inode 排序，尝试改善访问局部性
  -> full hash
  -> 生成 HashResult
```

小文件轨道的重点不是“少读”，而是“少浪费固定成本”。

因此，后续围绕小文件的优化主要包括：

```text
避免 buffer 无意义清零
减少不必要的大 buffer 分配
尝试 batch 降低任务数量
使用 thread_local buffer 复用临时缓冲区
```

---

## 大文件轨道

大文件轨道中的文件体积较大，直接 full hash 的成本很高。

因此，大文件不会立刻进入 full hash，而是先进入 partial hash 阶段：

```text
largeCandidates
  -> partial hash
  -> 找出 partial hash 仍然碰撞的文件
  -> 再进入 full hash
```

当前实现中，大文件 partial hash 采用 head + tail 的方式：

```text
读取文件头部一小段
读取文件尾部一小段
组合计算 partial hash
```

这样可以处理一种常见情况：多个大文件前缀相同，但尾部不同。如果只读头部，可能无法区分；加入尾部后，过滤能力更强。

大文件轨道的重点是：

```text
在保证正确性的前提下，
尽量减少不必要的完整文件读取。
```

需要注意，partial hash 不能作为最终重复判断依据。它只能作为 full hash 之前的过滤条件。最终是否重复，仍然必须由完整内容 hash 判断。

## 在整体流程中的位置

加入小文件与大文件分流之后，整体流程可以抽象为：

```text
扫描目录
  -> 收集 path / size / device / inode 等元信息
  -> 第 1 层：按 size 分组，大小唯一直接跳过
  -> 第 2 层：按 device / inode 折叠硬链接
  -> 第 3 层：按文件大小分成小文件轨道和大文件轨道

小文件轨道：
  -> 按 device / inode 排序
  -> full hash

大文件轨道：
  -> partial hash
  -> partial hash 碰撞后再 full hash

最终：
  -> 合并 full hash 结果
  -> 展开 hardlink follower
  -> 按 hash 分组
  -> 输出重复文件组
```

这一层的作用不是直接减少重复文件数量，而是为后续不同类型文件选择不同策略。

可以把它理解为一个策略分派层：

```text
小文件：
  固定成本主导
  优化方向是减少每个文件都会发生的开销

大文件：
  I/O 成本主导
  优化方向是减少无效 full hash
```

---

## 与后续优化的关系

这一层是后续两条优化主线的分界点。

第一条是小文件优化主线。

```text
小文件轨道
  -> full hash
  -> 优化每个文件都会发生的固定成本
```

这一条主线会解释为什么 `buffer` 分配和初始化会成为瓶颈，为什么避免 `std::vector<char>(bufferSize_)` 的清零成本能带来明显收益，以及为什么后续会尝试 batch 和 `thread_local` buffer。

第二条是大文件优化主线。

```text
大文件轨道
  -> partial hash
  -> 必要时再 full hash
```

这一条主线会解释为什么大文件不能简单直接 full hash，为什么 head + tail partial hash 可以显著减少无效 I/O，以及为什么真实重复的大文件最终仍然必须 full hash。

因此，这一层虽然代码很简单，但在整体优化结构中非常重要。它把原本统一的 hash 流程拆成了两条成本模型完全不同的执行路径。

---

## 小结

第 3 层优化的核心是小文件与大文件分流。

经过 size 过滤和 hardlink 折叠后，剩下的候选文件仍然具有不同的性能特征。小文件的主要问题是每个文件都会发生的固定成本，大文件的主要问题是 full hash 带来的大量 I/O。

因此，`dedup` 使用文件大小阈值将候选文件拆成小文件轨道和大文件轨道。小文件直接进入 full hash，后续重点降低固定成本；大文件先进入 partial hash，尽量避免不必要的完整文件读取。

这一层本身不是最终优化，而是策略分派层。它为后续“小文件固定成本优化”和“大文件 partial hash 优化”建立了清晰的结构边界。
