# dedup 项目背景

## 项目简介

`dedup` 是一个面向本地文件系统的文件去重工具，目标是扫描指定目录，快速找出内容相同的重复文件组。

文件去重的基本思路并不复杂：遍历目录读取文件信息 → 读取文件内容 → 计算文件哈希 → 然后把哈希相同的文件归为一组。但在真实场景中，直接对所有文件计算完整哈希并不是一个高效方案。因为真实目录中的文件通常具有不同的形态：可能有大量小文件，也可能有大文件，可能有大小不同的文件混杂在一起，也可能存在多个路径指向同一个物理文件的硬链接。

因此，`dedup` 的目标不是实现一个简单可用的文件去重工具，而是实现一个高性能的文件去重工具。它的核心思路是：先用低成本信息排除不可能重复的文件，再把昂贵的完整内容哈希留给真正需要验证的候选文件。

## 朴素去重流程

最直接的文件去重流程可以抽象为：

```text
扫描目录
  -> 收集文件路径、大小等元信息
  -> 对所有文件计算内容 hash
  -> 按 hash 分组
  -> 输出重复文件组
```

这个流程简单直接，但问题是它会对所有文件计算完整 hash，没有充分利用文件大小、inode 等低成本信息进行提前过滤。因此，当文件数量较多或文件体积较大时，这种方案会产生大量不必要的 I/O 和 hash 计算。

## 性能优化空间

并发执行是我首先考虑到的一个重要的性能优化方向。因为我们的场景是对大量文件进行处理，文件之间**没有计算依赖关系**，天然适合并发处理。线程池可以让多个文件的读取和 Hash 任务并行执行，重叠 I/O 等待和 CPU 计算。但因为我们的场景涉及到文件读取，因此我们需要考虑硬件设备因素：在 SSD 或 NVMe 这类随机访问能力较强的存储设备上，并发读取通常更容易带来收益；但在 HDD 上，过高并发可能导致频繁寻道或发生磁头抖动，反而不如顺序读吞吐。因此，线程池优化需要结合存储设备特性一起考虑。

除此之外，文件去重的性能瓶颈并不是单一的。不同文件形态下，主要成本也不同。

1. 对于大量小文件，单个文件的数据量很小，真正明显的往往不是 hash 计算本身，而是每个文件都会发生的固定成本，例如文件打开和关闭、buffer 分配与初始化、任务提交和结果管理等。当文件数量达到几千或几万个时，这些固定成本会被不断放大。

2. 对于大文件，主要成本来自 I/O 和完整内容哈希。尤其是当大文件数量增多时（甚至大小相同但内容不同），如果对所有大文件直接 full hash，就会读取大量最终没有必要读取的数据并进行 Hash 计算。这类无效 I/O 和 Hash 计算是大文件场景下最值得优化的部分。

3. 对于文件系统本身，还可以利用一些元信息减少重复工作。例如多个路径可能通过硬链接指向同一个物理文件，如果能够利用 `device / inode` 识别这种情况，就可以避免对同一份物理数据重复读取和重复 hash。

基于上面的分析，`dedup` 的优化可以分成四个方向：

1. **并发执行优化**：通过线程池并行处理文件任务，重叠 I/O 等待和 hash 计算。
2. **小文件优化**：降低每个文件都会发生的固定成本，这对于大量小文件场景比较重要。
3. **大文件优化**：减少不必要的完整文件读取和 hash 计算，这对于大文件场景比较重要。
4. **文件系统层优化**：利用 `device / inode` 等元信息减少重复工作。

最终目标是把原始的简单流程优化成一个更细的过滤漏斗：

```text
扫描文件
  -> 按 size 分组，大小唯一直接跳过
  -> 利用 device / inode 识别同一物理文件
  -> 小文件直接 full hash
  -> 大文件先做 partial hash
  -> 必要时再 full hash
  -> 输出重复文件组
```

这个漏斗的核心原则是：越靠前的阶段，使用越便宜的信息；越靠后的阶段，才执行越昂贵的操作。这样可以在保证去重结果正确性的前提下，尽量减少不必要的 I/O、hash 计算和固定开销。

> [!Important]
> 线程池不改变过滤漏斗的逻辑，它解决的是执行效率问题：如何让多个候选文件的读取和哈希计算更高效地并行执行。

## 项目总结

`dedup` 的性能优化可以概括为：先建立并发执行基线，再基于文件形态建立成本模型，并围绕这个模型设计分层过滤漏斗。

执行层通过线程池提升并行处理能力；小文件场景重点降低固定成本；大文件场景重点减少无效 I/O；文件系统层则利用元信息减少重复工作。

这个项目的重点不是某个单独的优化技巧，而是从 baseline 出发，分析不同场景下的性能瓶颈，设计针对性的 benchmark，再根据数据验证和修正优化方向。
