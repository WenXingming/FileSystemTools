# 第 1 层过滤：按文件大小进行候选筛选

线程池提升的是 hash 执行效率，但它并没有减少需要 hash 的文件数量。但如果所有文件都进入 hash 阶段，即使并发执行，仍然会做很多不必要的无效工作。因此接下来要思考的是：哪些文件根本不需要 hash？因此需要引入过滤漏斗：先用低成本信息排除不可能重复的文件。其中，第一个最直接、最可靠的过滤条件就是文件大小。

朴素的文件去重流程可以抽象为：

```text
扫描目录
  -> 收集文件路径、大小等元信息
  -> 对所有文件计算内容 hash
  -> 按 hash 分组
  -> 输出重复文件组
```

这个流程的问题是：它会对所有文件计算完整哈希，即使很多文件从大小上就已经不可能重复。因此，更合理的基础流程应该先利用文件大小进行过滤：

```text
扫描目录
  -> 收集文件路径、大小等元信息
  -> 按 size 分组
  -> 大小唯一的文件直接跳过
  -> 对 size 发生碰撞的候选文件计算内容 hash
  -> 按 hash 分组
  -> 输出重复文件组
```

大致代码如下：

```cpp
DuplicateReport DuplicateFinder::find_duplicates(const std::string& rootPath) const {
    const FileWalkResult walkResult = fileWalker_.collect_files(rootPath);

    // 第 1 重漏斗：按文件大小过滤
    const std::vector<FileInfo> candidates =
        find_size_collisions(walkResult.files);

    // 只对 size 发生碰撞的候选文件计算完整 hash
    const std::vector<HashResult> hashResults =
        compute_hashes(candidates);

    const std::vector<DuplicateGroup> duplicateGroups =
        find_exact_duplicates(hashResults);

    return assemble_report(walkResult, hashResults, duplicateGroups);
}
```

**文件大小是非常便宜且有效的过滤条件**。两个内容相同的文件大小一定相同；如果两个文件大小不同，就一定不是重复文件：

```text
content(A) == content(B)  =>  size(A) == size(B)
size(A) != size(B)        =>  content(A) != content(B)
```

通过 size 分组，可以避免对大量不可能重复的文件做无意义的内容读取和哈希计算：大小唯一或大小不同的文件可以安全跳过，不会漏掉真正重复的文件。

当然，size 过滤只能排除大小不同的文件。如果大量文件大小相同，那么它们仍然会进入后续 hash 阶段。例如很多 4KiB 小文件、固定大小切片文件、日志块、缓存块等，都可能在 size 维度发生大量碰撞。这个时候，性能瓶颈会继续转移到 full hash 阶段。所以当前的 size 过滤只是第一层过滤，后续我们将引入更多优化方案。
